# 360doegaku AI開発共通テンプレート

## 0. 作業開始前の確認

作業を開始する前に、以下のドキュメントを確認すること。

- PROJECT.md
- ARCHITECTURE.md
- ROADMAP.md

今回の作業内容が、プロジェクト全体の設計方針と矛盾していないことを確認してから実装を開始する。

---

## 1. 作業名

ここに今回の作業名を記入する。

例：

フェーズ2-4：自動回転の停止

---

## 2. 目的

今回の作業で達成したい内容を記入する。

例：

球体の自動回転を停止し、360度空間の基礎表示を静止状態にする。

---

## 3. 現在の状態

作業前の状態を記入する。

例：

- 球体の内側が表示されている
- ワイヤーフレームが右から左へ自動回転している
- Three.js関連コードは役割ごとに分割済み

---

## 4. 修正対象

変更してよいファイルを明記する。

例：

- src/renderer/src/renderer.ts

指定されていないファイルは変更しないこと。

---

## 4-1. 修正範囲

指定したファイルのみ変更すること。

他のファイルにも修正が必要と思われる場合は、勝手に変更せず、その理由を報告すること。

変更が必要な理由と、変更予定ファイルを事前に報告すること。

---

## 5. 実装内容

今回実装する内容を具体的に記入する。

例：

- アニメーションループ内の球体回転処理を削除する
- レンダリング処理は維持する
- 球体の内側表示は維持する

---

## 5-1. 実装方針

- 既存コードは、目的達成に必要な場合のみ削除すること
- 削除した場合は、削除理由を必ず報告すること
- 指示がない限り、公開API・関数シグネチャを変更しないこと
- 既存のコーディング規約（命名規則・インデント・セミコロンの有無・クォーテーション・改行形式など）を維持すること
- 既存のコメントスタイル・コメント言語（日本語）を維持すること
- 推測で実装を続行せず、問題があれば作業を停止して報告すること
- 1回の作業で変更が約200行を超える見込みの場合は、事前に相談すること
- 複数ファイルへ変更が及ぶ場合は、実装前に変更予定ファイル一覧と変更理由を報告すること
- 既存のクラス・関数・責務を優先して利用し、同じ役割を持つ処理を新規作成しないこと
- 既存コメントは、内容が誤っている場合を除き削除しないこと

## 6. 必須条件

必ず守る条件を記入する。

例：

- TypeScriptエラーを発生させない
- 現在の見た目を回転停止以外は変更しない
- import/exportを壊さない
- npm run typecheckを通す
- npm run lintを通す

---

## 7. 変更禁止

今回変更してはいけない内容を記入する。

例：

- 新機能の追加
- OrbitControlsの追加
- マウス操作の追加
- ファイル構成の変更
- 変数名の変更
- 大規模なリファクタリング
- 指示のないコード整理
- 指示のないフォーマット変更
- 指示のない命名変更
- package.jsonの変更
- CSSの変更
- ライブラリの追加
- コメントの大量追加

---

## 7-1. 実装開始前の自己確認

実装を開始する前に、以下を確認すること。

- 「現在の状態」を読んだ
- 「目的」を読んだ
- 「実装内容」を読んだ
- 「変更禁止」を読んだ

そのうえで、以下を日本語で簡潔に要約すること。

### 今回実装する機能

- 今回の「目的」と「実装内容」に記載されている機能のみを記入する

### 今回実装しない機能

- 「現在の状態」ですでに実装済みとされている機能
- 「変更禁止」に記載されている機能
- 今回の「目的」と「実装内容」に記載されていない機能

### 変更予定ファイル

- 変更するファイル名
- 各ファイルを変更する理由

「今回実装する機能」と「今回実装しない機能」を区別できない場合、
記載内容に矛盾がある場合、または判断に迷う場合は、
実装を開始せず、日本語で質問すること。

すでに実装済みと記載されている機能を、
今回の実装対象として再実装しないこと。

今回の目的と無関係な機能を追加・修正しようとしている場合は、
指示を誤読している可能性があるため、実装を停止して内容を再確認すること。

上記の要約および変更予定ファイルを報告した後は作業を停止すること。

ユーザーから「実装開始してください」などの明示的な指示を受けるまで、
コードの追加・削除・修正は一切行わないこと。

## 9. 作業結果の報告

作業後は、以下を簡潔に報告すること。

- 変更したファイル
- 変更内容
- 実行した確認コマンド
- 確認結果
- 変更理由
- 削除したコード（ある場合）と削除理由
- GUI環境の制約など、確認できなかった項目
- git diff の確認結果
- git status の確認結果
- 未確認事項については推測を書かず、「未確認」と事実のみを報告すること

---

## 10. 共通ルール

- 一度の作業では一つの目的だけを扱う
- 指示されていない機能は、便利そうであっても追加しない
- 指示されていないファイルを変更しない
- 現在の動作を必要以上に変更しない
- 不明点や曖昧な仕様がある場合は推測せず、作業前に日本語で質問する
- 既存コードの命名・構造・コーディング規約を尊重する
- 既存のコメントスタイル・コメント言語（日本語）を維持する
- 最小限の変更で目的を達成する
- 作業後に差分を確認する
- git diff と git status を確認する
- 目的を満たしているか自己レビューを行う
- テストやビルドが失敗した場合は、原因を報告し未確認事項を明記する
- 動作確認できた項目と、GUI等の制約で未確認の項目を区別して報告する
- エラーがある場合は隠さず報告する
- 同じ機能を持つ処理を重複して作成しない
- 新しい実装は既存設計へ自然に統合する
- 必要以上に新しいファイル・関数・クラスを増やさない
- 確認できていない内容は推測せず、「未確認」と事実のみを報告すること

## 12. コメント記述ルール

コードへ日本語コメントを追加する際は、以下のルールに従うこと。

- 新規ファイルには、ファイル先頭にそのファイルの役割を説明するコメントを付ける
- 関数には、その関数が何のためにあるかを説明するコメントを付ける
- 複雑な処理には、「何をしているか」だけでなく「なぜ必要なのか」も書く
- 処理のまとまりごとに、区切りが分かるようコメントを書く
- 将来の実装予定は TODO、注意事項は NOTE
  を使って書く（現在の設計資料から確認できない予定を勝手に作らない）
- コメントは日本語で、短く分かりやすい文章にする
- 既存のコメントスタイルとコメント言語（日本語）を維持し、英語コメントなどを混在させない
- 専門用語を使う場合は、初心者でも理解できるように補足する
- コードを日本語へ直訳しただけの不要なコメントは避ける
- コメントを追加・修正する作業では、処理内容や動作を変更しない

### 12-1. コメントは「What」より「Why」を優先する

コメントを書く際は、「何をしているか（What）」だけではなく、
「なぜその処理が必要なのか（Why）」を優先して説明してください。

コードを読めば分かる内容をそのまま日本語へ直訳したコメントは避け、
設計意図・理由・目的・背景が伝わるコメントを記述してください。

優先順位は次のとおりです。

1.  なぜその処理が必要なのか（Why）
2.  その処理の目的（Purpose）
3.  何をしているか（What）
4.  どのように実装しているか（How）

How
はコードを読めば理解できるため、コメントでは必要最小限にしてください。

【悪い例】

// カメラを作成する const camera = new THREE.PerspectiveCamera(...);

→ コードを見れば分かるため、説明としての価値が低い。

【良い例】

// 360度空間を球体の内側から見渡せるようにするため、 //
カメラを球体の中心へ配置する。 const camera = new
THREE.PerspectiveCamera(...);

【悪い例】

// 球体をシーンへ追加する scene.add(sphere);

【良い例】

// シーンへ追加することで、レンダラーが球体を描画対象として認識できる。
scene.add(sphere);

【悪い例】

// レンダリングする renderer.render(scene, camera);

【良い例】

// 現在のシーンをカメラ視点から画面へ描画する。 //
この処理は毎フレーム実行され、画面更新の最後に呼び出す。
renderer.render(scene, camera);

コメントは「コードを読めば分かること」ではなく、
「コードだけでは分からない設計意図」を補足することを目的としてください。

## 13. パフォーマンス実装ルール

リアルタイム描画アプリであることを考慮し、以下を守ること。

- 毎フレーム不要なオブジェクトを生成しない
- render loop内で不要なnewを行わない
- 同じ計算は可能な限り再利用する
- GPUリソースは不要になったらdispose()する
- パフォーマンスを著しく低下させる実装を行わない
- パフォーマンスへ影響する変更を行った場合は、その理由を報告する

---

## 14. AI実装方針

- 推測で実装しない
- 曖昧な仕様は推測せず、日本語で質問する
- 不明点は勝手に補完しない
- 設計資料を優先する
- 過去の実装方針と矛盾しないことを確認する
- 指示にない機能は、便利そうであっても追加しない
- 既存のコーディング規約を維持する
- 既存のコメントスタイル・コメント言語（日本語）を維持する
- 目的外の変更が必要と判断した場合は実装せず、その理由を日本語で説明する
